用 Lit 写一个网页音乐播放器，支持本地音频文件播放。

主界面分三个区域：
- 顶部是当前播放信息（封面图、歌曲名、艺术家、专辑、进度条、时间显示）
- 中间是播放控制（上一首、播放/暂停、下一首、循环模式、随机播放、音量滑块、静音按钮）
- 底部是播放列表（可折叠）

播放列表：
- 用户拖拽本地音频文件到播放列表区（多文件）或者点"添加文件"按钮选
- 支持的格式 mp3、wav、ogg、flac、m4a（不支持的格式弹错误提示，不入列表）
- 列表项显示：序号、歌曲名（从文件名解析或者从音频元数据 ID3 读）、时长、双击播放
- 列表项可拖动排序，右键菜单有：从列表移除、在文件管理器中显示、查看元数据
- 列表支持多选（Shift 范围选、Ctrl 点选），多选后可以批量删除或者加入下一播放队列
- 列表可保存为"歌单"（多个歌单切换），歌单数据存 localStorage

播放核心：
- 用 HTMLAudioElement 播放
- 进度条可点击/拖动跳转
- 音量滑块 0-100，静音时图标变化
- 循环模式：列表循环（默认）、单曲循环、不循环
- 随机播放：不重复直到列表里所有歌都播过一遍
- 播放结束自动切下一首（按列表循环或随机规则）
- 用户切换页面（visibilitychange 隐藏）时音频继续播（不要暂停）

歌词同步：用户可以给当前播放的歌曲加载 LRC 歌词文件（拖到播放器上），解析 LRC 时间标签，按播放进度高亮当前行，点击歌词行能跳转到对应时间。LRC 解析要支持 [00:01.23] 这种标准格式和 [00:01.234] 三位毫秒的格式。

主题切换：浅色/深色/跟随系统三档，主题色用 CSS 变量。

元数据读 ID3v2 标签（用浏览器内置的 AudioContext.decodeAudioData 不能读 ID3，要写一个简单的 ID3v2 parser 或者用现成的轻量库；或者只用文件名当标题也行，但要写明)。

至少 3 个文件：player 组件、playlist 组件、lyrics 组件分开。第三方依赖尽量少，不要 audio.js、howler 之类。

实际工作里还会遇到这些情况：音频文件大于 2GB 时分片读取（不全加载到内存）。播放列表拖动排序实时更新播放索引。歌词滚动到当前行时高亮（前后 2 行半透明）。歌词点击跳转精确到秒（不丢失精度）。主题切换时所有界面元素跟随变色。播放进度记忆（用户上次听到 1:23，下次打开从 1:23 继续播，10 秒内）。文件读取错误时显示原因（"文件格式不支持"等）。浏览器拒绝自动播放时显示提示（让用户点页面触发播放）。元数据 ID3 解析失败降级用文件名。播放列表导入 m3u 文件（m3u8 也支持）。

补段用户场景：用户用这个播放器听本地音乐库（自己下载的 MP3 文件存硬盘上）。他拖一整个专辑文件夹到应用，自动按文件名排序加入列表（按音轨号 01-12 这种）。ID3 元数据读了歌曲名、艺术家、专辑、封面（如有），用这些信息展示在主界面。

他有时候会切到浏览器其他 tab 看文档，音频继续播。切回来播放器界面要保持（不能重新加载）。他听某首歌听到 1:23 喜欢这个片段想分享出去——导出当前播放进度（带时间戳）。

技术细节：HTMLAudioElement 的 ended 事件触发后判断是用户停止还是播放结束（用户停止不切下一首）。currentTime 设置跳转时浏览器有缓冲延迟（不要连续多次设置，用 debounce）。播放列表 1000+ 首歌时 DOM 渲染要虚拟列表（不然滚动卡）。多选拖拽排序时计算新位置用索引而不是坐标（避免像素误差）。

播放器是 PWA 应用，所以要先注册 service worker 缓存静态资源，离线状态下打开也能看播放列表（播放需要联网拉流）。播放列表用 IndexedDB 存储歌曲元数据（标题、艺术家、专辑封面 URL、时长），封面按 lazy 加载，列表滚动到底才加载更多。Web Audio API 处理播放、暂停、跳转、音量控制。

歌词同步：LRC 格式的歌词按时间戳分行渲染，当前播放时间命中哪行就高亮哪一行，行间滚动要平滑过渡（CSS transform translateY）。点击歌词行能跳转到对应时间点。歌词文件按歌曲 ID 缓存到 localStorage。

播放模式：顺序播放、单曲循环、列表循环、随机播放四种模式切来切去。音量滑块要支持记忆（每次开播恢复到上次音量）。最后播放的歌曲和位置在页面刷新后能自动恢复（localStorage + 启动时 seek）。

实际工作流里还会有这些场景。一个本地音乐文件被加进播放器后走"待播放-正在播放-已暂停-已播完-播放失败"五个状态，外加"被标记为不可用"（文件丢失或损坏后变灰但保留在列表里）。"待播放"在播放队列里等待，"正在播放"被加载到 audio element 并开始播，"已暂停"用户手动暂停后保持当前位置，"已播完"走完了时长触发 ended 事件切下一首，"播放失败"是解码错误或者文件读不出来（用户能在文件管理器里检查原文件）。"被标记为不可用"是定时（每 5 分钟）扫一次列表里所有文件，对应 file:// 路径或者 IndexedDB 里存的 blob handle 是否还能拿到，丢失的标灰但保留用户打的标签和歌单位置。歌单本身也走"草稿-已发布-已归档"——草稿歌单用户在编辑中可任意删歌，已发布歌单（被分享出去或者被多人订阅）不能再删歌只能"标记不再收录"（歌还在但列表里不显示）。

多角色视图：管理员（设备所有者）能导入/删除音乐文件、创建/编辑歌单、设置全播放器参数（默认音量、循环模式、跨设备同步开关）。普通用户（家庭成员/合租室友）能创建自己的歌单、用自己的播放历史、不能删管理员的文件——管理员的文件对普通用户显示为"只读"图标（带锁）。访客（朋友临时用一下）只能用当前正在播放的那首歌和最近 10 首历史，不能创建歌单不能改设置，访客会话 30 分钟无操作自动退出。儿童模式（独立 PIN 锁）只显示家长标记为"儿童友好"的歌曲（家长在元数据里加 flag），音量上限 60%，不能访问网络电台。

时间与配额：免费版单设备最多 200 首歌（超过提示"升级到订阅版"），订阅版（设置里勾"已订阅"模拟）无上限。播放器每首歌听完后会写一条"播放历史"——包含歌 ID、听的时间、听完没听完、当时音量、跳过了哪些片段。月底最后一天 23:59 弹"本月听歌报告"——本月听歌总数、听的时间分布（按小时柱状图）、最爱的歌手 Top 5、新发现歌曲数（第一次听的歌）。每首歌超过 30 天没播放会被自动从"最近播放"列表里下移（移到"尘封歌单"分类里），从尘封里手动播放一次又回到"最近播放"。播放历史超过 1000 条自动压缩（旧的多条合并成"X 月听了 Y 首歌"汇总记录）。

并发与数据一致性：用户在两个 tab 打开播放器同时播不同歌时，第二个 tab 启动时通过 BroadcastChannel 检测到已有 tab 在播，自动变"远程控制模式"——只显示当前播放信息和控制按钮（上一首/暂停/下一首/音量），播放实际在第一个 tab 进行，避免两路音频打架（浏览器只允许一路 audio 上下文活跃）。文件被删/移动时所有引用该文件的歌单都标记"含失效歌曲"显示橙色感叹号，用户可以选择"从所有歌单移除"或者"重新定位文件"（手动选新路径）。导入歌单 JSON 时检测 ID 冲突——冲突时弹"覆盖/合并/重命名"三个选项，合并模式下导入的歌曲追加到现有歌单尾部（同名歌曲去重按文件 hash）。导入文件带 schemaVersion 版本号，跨大版本（比如 v1 歌单元数据格式 v2 不识别）拒绝导入并提示用户。导出时除了歌单还要带"播放统计快照"——听歌次数、上次听的时间、用户评分（1-5 星），导入时可以选择"仅导入歌单结构"或"同时导入统计数据"。
